## 問題紀錄

### JSON / Schema 問題

9 次測試中，8 次成功回傳可解析 JSON 並通過 `validateReportData`。未觀察到 validate errors。第 8 次未進入 JSON / Schema 驗證，因為 Gemini API 回傳 HTTP 429。

### 報告內容問題

內容品質尚未逐份人工審閱。技術測試只確認 JSON 結構、validate、normalize 與預覽流程。正式判斷 Gemini 是否可作為 provider 前，需要逐份檢查語氣、風險判斷與是否過度承諾。

### 產品樣貌問題

待人工逐份檢查 `productShape.format`、`productShape.userFlow`、`productShape.userSees`、`productShape.firstVersionLook` 是否足夠具體且可想像第一版產品。

### 風險判斷問題

測試摘要顯示多數成功案例仍需人工複查風險涵蓋。重點是寵物健康 / 醫療責任、監控隱私與串接風險、以及 AI 副業啟動包差異化不足是否有被明確指出。

### 其他問題

第 8 次測試遇到 Gemini API HTTP 429。若未來進入正式 provider 候選，至少需要加入 retry、退避等待、錯誤提示與 rate limit 防護。

## 最後決策

Gemini 需要修 Prompt 後再測。

理由：

- 技術流程已達到 8/9 可預覽，接近暫定通過標準。
- 但出現 1 次 HTTP 429，需要補 PoC 層級的重試或節流策略。
- 內容品質欄位仍待人工逐份審查，尚不能判定 Gemini 可進入正式 provider 候選。

## 第二輪問題紀錄

### 429 / rate limit

10 秒間隔重測後，9 次測試沒有出現 `GEMINI_RATE_LIMITED`。429 防護在這次測試中未被觸發。

第 7、8、9 次出現 `GEMINI_TEMPORARY_ERROR`，代表 Gemini 暫時無法回應，且 retry/backoff 後仍失敗。

### JSON / Schema

5 次成功預覽都可解析 JSON、通過 validate，warnings 為 0，normalize 後 score / scoreLabel 一致。

第 2 次 Gemini 有回應，但 validate 失敗：`pricingTiers 必須是 array`，因此無法預覽報告。

第 7、8、9 次未取得 Gemini 報告，因此未進入 JSON / Schema 驗證。

### 人工內容複查

成功的寵物飼料報告都有提到健康、營養、醫療責任、資料維護與免責，也避免直接承諾推薦效果。第 1 次產品樣貌偏 Email 報告，需小修；第 3 次可接受。

成功的 AI 寵物監控摘要報告都能指出技術、隱私、平台串接、長期維護與 AI 誤判風險。第 4 次偏概念驗證 Landing Page，第 5、6 次偏人工服務或單次報告，作為保守驗證方向可接受，但若要作為產品報告仍需小修。

AI 副業啟動包本輪沒有成功產出內容，因此無法完成內容品質判斷。

## 第二輪決策

Gemini 需要修 Schema 後再測。

理由：

- 10 秒間隔重測後，HTTP 429 次數為 0，rate limit 不再是本輪主要問題。
- 第二輪 9 次測試只有 5 次成功預覽，低於至少 8 次成功預覽的通過標準。
- 第 2 次發生 `pricingTiers 必須是 array`，表示 Gemini 仍可能輸出不符合 schema / validate 期待的欄位格式。
- 第 7、8、9 次出現 `GEMINI_TEMPORARY_ERROR`，穩定性仍不足。
- 成功報告內容多數冷靜保守，且沒有明顯過度承諾；但部分產品樣貌偏 Landing Page 或人工服務，仍需小修。
