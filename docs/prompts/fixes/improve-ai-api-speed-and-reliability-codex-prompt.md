# Codex 任務提示詞：改善 AI API 速度慢與偶發錯誤問題

你現在要在既有的「副業點子冷靜評分器」專案中，改善 AI API 回應速度慢、偶發錯誤、使用者等待體驗不佳的問題。

## 一、任務目標

這次不是重做 AI 架構，也不是更換供應商。

目標是先用低風險方式提升穩定性與使用者體驗：

1. 加入 API timeout
2. 加入有限次數 retry
3. 改善錯誤訊息
4. 避免重複送出
5. 保留既有 loading 狀態
6. 若 AI 失敗，提供清楚的下一步
7. 不影響一般報告、深度報告、回饋表單、使用次數限制、報告紀錄頁

---

## 二、目前問題

使用者反映：

```text
AI API 速度很慢，有時候會錯誤
```

可能原因包含：

1. Gemini API 回應時間不穩定
2. 請求內容太長
3. 模型偶發回傳格式不穩
4. 前端等待時沒有足夠提示
5. 使用者可能重複點擊送出
6. API 沒有 timeout，導致等待過久
7. 429 / 500 / 503 等錯誤沒有被友善處理

---

## 三、這次優先改善項目

### 1. API timeout

請在呼叫 AI API 時加入 timeout。

建議：

```text
一般報告 timeout：45 秒
深度報告 timeout：75 秒
```

如果目前 Cloudflare Functions / API route 有平台限制，請依照實際可行時間調整，但不要讓使用者無限等待。

逾時時回傳友善錯誤：

```text
AI 回應時間過久，這次沒有成功完成。請稍後再試，或縮短輸入內容後重新送出。
```

---

### 2. retry 機制

只針對可重試錯誤 retry。

建議 retry 條件：

```text
429
500
502
503
504
network error
timeout
```

不要對以下錯誤 retry：

```text
400
401
403
JSON schema 明顯錯誤
使用者輸入缺失
```

retry 次數：

```text
最多 1 次 retry
```

不要做 3 次以上，避免 API 成本暴增。

retry 間隔：

```text
800ms ~ 1500ms
```

可用簡單固定延遲，不需要複雜 exponential backoff。

---

### 3. 前端避免重複送出

使用者送出後：

1. 按鈕 disabled
2. 顯示 loading
3. 不允許連續點擊造成多次 API 呼叫
4. 若失敗，才恢復按鈕

如果目前已經有 disabled，請檢查一般報告與深度報告都確實有效。

---

### 4. Loading 文案分階段

保留原本 loading，但請加上分階段提示。

建議：

0 秒：

```text
正在分析你的副業點子...
```

15 秒後：

```text
AI 正在整理風險與 MVP 建議，這可能需要一點時間。
```

30 秒後：

```text
目前 AI 回應較慢，請不要關閉頁面或重複送出。
```

若深度報告可使用不同文案：

```text
深度報告內容較長，產出時間通常會比一般報告久。
```

---

### 5. 錯誤訊息分類

請不要只顯示 generic error。

至少分成以下幾類：

#### 429 / quota / rate limit

```text
目前 AI 使用量暫時達到限制，請稍後再試。
```

#### timeout

```text
AI 回應時間過久，這次沒有成功完成。請稍後再試，或縮短輸入內容後重新送出。
```

#### JSON parse / schema error

```text
AI 回傳格式不完整，系統無法正確產生報告。請重新產出一次。
```

#### unknown error

```text
報告產出失敗，請稍後再試。如果持續發生，請截圖錯誤訊息回報。
```

---

### 6. 不要在失敗時扣除使用次數

請檢查目前使用次數限制邏輯。

要求：

1. API 成功產出報告後，才扣除一般報告或深度報告次數
2. timeout、429、500、JSON parse 失敗，不應扣除次數
3. 使用者重複點擊不應造成多次扣除

如果目前架構是送出前就扣次數，請改成成功後才扣。

---

### 7. 保留原始錯誤給開發者看

使用者前台顯示友善錯誤。

console / server log 保留比較完整的錯誤資訊，例如：

```text
status code
provider error message
mode: free / deep
timeout or retry count
```

但不要把 API key、完整環境變數、敏感資訊顯示在前台。

---

## 四、降低 AI 回應時間的輕量優化

請檢查 prompt 是否有不必要的冗長內容。

這次只做小幅整理，不要重寫整套 prompt。

原則：

1. 一般報告 prompt 保持短一點
2. 深度報告才放完整分析要求
3. 不要在一般報告中要求輸出過多長文
4. JSON schema 說明保留必要欄位即可
5. 避免要求 AI 同時產出太多重複段落

如果有明顯重複內容，可以整理。

---

## 五、可選但建議：產生失敗時保留使用者輸入

如果報告產出失敗，不要清空表單或使用者輸入。

使用者應可以直接按「重新產出」。

可顯示：

```text
你可以直接重新產出，不需要重新填寫內容。
```

---

這次不要做：

```text
更換 AI 供應商
串多模型 fallback

資料庫 job queue
會員系統
Email 通知

付款功能

```

這些等產品驗證後再做。

## 八、驗收標準

完成後請確認：

1. 一般報告送出後，按鈕會 disabled，不能連續點擊
2. 深度報告送出後，按鈕會 disabled，不能連續點擊
3. AI API 超時會顯示友善錯誤
4. 429 / quota 類錯誤會顯示友善錯誤
5. JSON parse 失敗會顯示友善錯誤
6. 可重試錯誤最多 retry 1 次
7. 失敗不扣使用次數
8. 成功才扣使用次數
9. 失敗後使用者輸入不會消失
10. `npm run build` 通過
11. 不影響既有報告紀錄頁、回饋表單、一般報告、深度報告
